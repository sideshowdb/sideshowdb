# RFCs (requests for comment)

Place in-flight design proposals here as Markdown. An RFC should:

- State the problem and constraints in plain language.
- List options and trade-offs.
- Name a **preferred** option and what would need to be true to accept it.

When an RFC is accepted, either:

- Promote it to an ADR under [`../adrs/`](../adrs/) (short, decision-focused), **or**
- Fold its content into specs under `docs/development/specs/` if it is
  purely normative user-facing behavior.

Withdrawn RFCs can stay in this folder with `Status: Withdrawn` in the header.
